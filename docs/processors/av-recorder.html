av_recorder

Pauses the pipeline to collect a user recording (webcam/mic capture), then returns it as a source clip for downstream steps.

When to use

Use av_recorder when you want to insert a live speaker recording into the pipeline at a specific point. The processor blocks until the user provides a recording file — either via the web UI (when RecordingHandler is wired) or by entering a file path at the terminal (CLI fallback).

An optional script input is used to display teleprompter text during recording and to generate a brief AI shot concept (when an LLM is available).

YAML Example

- id: record_intro
  processor: av_recorder
  input:
    - id: script
      file: .puffinship/scripts/intro.txt   # optional
  params:
    rotation: 90   # optional: 90, 180, or 270 — overrides web upload rotation, stable across reruns
  output:
    timeline: main

Inputs

ID Source Required Description
script file no Script text file displayed to the user during recording and used for AI shot concept generation

Params

Param Type Default Description
rotation select: 0, 90, 180, 270 "0" Clockwise rotation to apply to the recording in degrees. Valid values: 90, 180, 270. When set, this overrides the rotation supplied via the web upload form, ensuring reruns use the same rotation without requiring a new recording. The rotation is part of the cache key — changing it invalidates the cache.

Web Form Parameters

When using web mode (via RecordingHandler), the following optional parameters may be supplied with the recording upload:

Parameter Type Description
rotation int Clockwise rotation in degrees (90, 180, or 270). Applied as a fallback when no rotation YAML param is set. The YAML param takes precedence.

Output

EditDecisions — one edit decision with Source set to the absolute path of the recording file. In and Out are both nil (full file).

Behavior

  1. If a script input is provided, the script text is read from disk.
  2. If an LLM is configured and a script is present, a 1-2 sentence shot concept is generated.
  3. Default concept (used when no LLM or script): "Record yourself reading the script to camera. Speak naturally."
  4. Web mode: calls ctx.RecordingHandler (blocking) with the step ID, concept, and script. The handler is responsible for presenting the recording UI and returning the saved file path.
  5. CLI fallback: prints the concept and script to stdout, then reads a file path from stdin.

Steps

  1. Remux to MP4: Browser-recorded WebM files lack a duration header, which causes ffprobe to fail in downstream steps. The processor re-muxes the recording to MP4 using ffmpeg (-c:v libx264 -preset fast -crf 18 -c:a aac -avoid_negative_ts make_zero -movflags +faststart). The -avoid_negative_ts make_zero flag shifts all streams by the same offset so A/V sync is preserved — do not replace this with per-stream setpts/asetpts filters, which independently reset each stream to 0 and destroy the relative timing.
  2. Apply rotation: The final rotation is determined by: YAML rotation param (if non-zero) → web upload form rotation field → no rotation. A second ffmpeg pass applies the appropriate filter:
    • 90°: transpose=1 (swaps dimensions to portrait for landscape input)
    • 180°: hflip,vflip (horizontal and vertical flip)
    • 270°: transpose=2 (swaps dimensions) The rotated file is written as <name>_rotated.mp4 and used as the output.

Notes

  • The processor works with or without an LLM — concept generation is best-effort and non-fatal.
  • The processor has no required inputs; the script input is fully optional.
  • Cache behaviour is identical to other processors — the step is cached after the recording path is returned. Clear the step cache to re-record.
